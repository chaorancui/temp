# C++ 知识点总结

## 注册机制

### 业界主流写法

业界最常用的设计思想是"反射式/工厂自动注册"。利用**全局/静态变量的初始化阶段**自动执行注册逻辑，同时为了代码清爽用宏封装静态变量注册代码。（最经典、最常用）

**典型场景**：根据字符串动态创建不同的处理器（如游戏技能、消息解析器）

```cpp
#include <iostream>
#include <string>
#include <unordered_map>
#include <memory>
#include <functional>

// 1. 产品基类
class MessageProcessor {
public:
    virtual ~MessageProcessor() = default;
    virtual void Process() = 0;
};

// 2. 全局注册表工厂（单例模式）
class ProcessorFactory {
public:
    using Creator = std::function<std::unique_ptr<MessageProcessor>()>;

    static ProcessorFactory& Instance() {
        static ProcessorFactory instance;
        return instance;
    }

    // 注册函数
    bool Register(const std::string& name, Creator creator) {
        registry_[name] = creator;
        return true;
    }

    // 创建函数
    std::unique_ptr<MessageProcessor> Create(const std::string& name) {
        auto it = registry_.find(name);
        if (it == registry_.end()) return nullptr;
        return it->second(); // 执行Lambda创建对象
    }

private:
    ProcessorFactory() = default;
    std::unordered_map<std::string, Creator> registry_;
};

// 3. 辅助注册类（利用构造函数触发注册）
class RegisterHelper {  // 或者叫 Registrar
public:
    RegisterHelper(const std::string& name, ProcessorFactory::Creator creator) {
        ProcessorFactory::Instance().Register(name, creator);
    }
};

// 4. 具体产品A
class LoginProcessor : public MessageProcessor {
public:
    void Process() override { std::cout << "Processing Login Message...\n"; }
};

// 核心：利用全局静态变量在 main 执行前完成注册
static RegisterHelper g_login_register("login", []() {
    return std::make_unique<LoginProcessor>();
});

int main() {
    // 业务使用：直接根据字符串创建，无需修改 main 或工厂类的代码
    auto processor = ProcessorFactory::Instance().Create("login");
    if (processor) {
        processor->Process();
    }
    return 0;
}
```

**宏定义封装**：为了让业务层写起来更爽，业界通常会用宏（Macro）把上面的静态变量注册代码封装起来，隐藏细节。

```cpp
// 注册宏定义
#define REGISTER_PROCESSOR(CLASS_NAME, KEY)\
    static RegisterHelper g_register_##CLASS_NAME(KEY, []() {\
        return std::make_unique<CLASS_NAME>();\
    });

// 业务层只需要这样写：
class LogoutProcessor : public MessageProcessor {
public:
    void Process() override { std::cout << "Processing Logout Message...\n"; }
};

// 一行代码搞定注册
REGISTER_PROCESSOR(LogoutProcessor, "logout");
```

### 常见踩坑点

在C++中玩自动注册，由于涉及底层链接期和初始化顺序，稍有不慎就会遇到极其隐蔽的Bug。以下是业界最常踩的 3 个大坑：

<font color=blue><b>坑 1：</b></font>静态初始化顺序地狱（Static Initialization Order Fiasco）

- **现象**：程序启动时直接崩溃（Segmentation Fault），或者注册表莫名其妙是空的。

- **原因**：C++ 标准规定，**不同编译单元（不同的 .cpp 文件）的全局/静态变量初始化顺序是未定义的**。如果你的"注册表工厂单例"和"业务层静态注册变量"在不同的文件里，有可能业务层去调用 `Register()` 时，工厂的 `unordered_map` 还没初始化呢！这就导致了往一个未初始化的内存写数据（崩掉），或者写完后工厂才初始化（数据被覆盖清空）。

- **解决措施**：使用 **Meyers 单例模式**（如上面代码所示）。在工厂的 `Instance()` 函数内部使用 `static ProcessorFactory instance;`。C++ 保证了局部静态变量在第一次被访问时才初始化，这确保了注册表一定比注册动作先诞生。

<font color=blue><b>坑 2：</b></font>静态库（.a / .lib）链接时注册失效（最难排查的坑）

- **现象**：如果把具体的产品类（比如 `LoginProcessor`）写在一个单独的**静态库**里，然后主程序去链接这个静态库。程序运行的时候，发现怎么都找不到 "login" 注册项。

- **原因**：**编译器的优化策略**。链接器在链接静态库时，是按需链接的。如果主程序中没有**显式**调用过 `LoginProcessor` 里的任何符号（函数或变量），链接器会认为这个文件里的代码"没被用到"，为了减小体积，直接把这个 `.o` 文件整个给丢弃（Strip）掉。既然文件被丢弃了，里面的全局静态注册变量自然也就没有执行。

- **解决措施**：
  1. **强行链接（平台相关）**：
     - Linux/GCC: 链接时加上 `-Wl,--whole-archive -lYOUR_STATIC_LIB -Wl,--no-whole-archive`。
     - Windows/MSVC: 链接时加上 `/WHOLEARCHIVE:YOUR_STATIC_LIB.lib`。 这会告诉链接器：别自作聪明，把这个库里的东西全部打包进来。
  2. **动态库化**：将静态库改为动态库（.so / .dll），动态库默认会保留这些符号。
  3. **显式桥接**：在主程序里手动调用一下该静态库里的一个空函数，强行建立引用关系（略显低效，破坏了解耦）。

<font color=blue><b>坑 3：</b></font>多线程并发注册导致 Crash

- **现象**：程序启动或动态加载插件（Dlopen）时，偶发性崩溃。

- **原因**：如果注册行为发生在多线程环境下（例如在某些动态链接库被异步加载时），多个线程同时往工厂的 `std::unordered_map` 里面 `insert`，而 C++ 标准容器不是线程安全的，导致红黑树或哈希表结构破坏。

- **解决措施**：
  - 如果注册**全部发生在 `main` 函数之前**，通常是单线程的，无需加锁。
  - 如果是支持**运行时动态注册（如动态加载插件）**，必须在工厂类的 `Register` 函数中加上互斥锁（`std::mutex`）：

    ```cpp
    bool Register(const std::string& name, Creator creator) {
        std::lock_guard<std::mutex> lock(mutex_); // 保证多线程安全
        registry_[name] = creator;
        return true;
    }
    ```

<font color=blue><b>坑 4：</b></font>单例模式失效

> 可能因编译器符号可见性（Visibility）与动态链接库（DSO）机制结合，导致的“单例模式失效”的底层大坑。

- **现象**：写代码时所有注册类都放在 `.h` 头文件中。调试时发现可以正确注册，但是获取不到注册的信息。

- **原因**：因为编译器参数和符号隐藏，你以为的"全局唯一单例"，在动态库的边界上被"斩断"成了多个独立的单例。具体如下：
  - 什么是 `TU`（编译单元）和 `inline` 副本

    在 C++ 中，每个 `.cpp` 文件连同它 `#include` 的头文件，被编译器处理时都称为一个**编译单元（Translation Unit, TU）**。 当你在头文件里写了一个 `inline static xxxRegistry& GetInstance()` 并且里面包含 `static xxxRegistry instance;` 时：
    - 编译器在编译 `A.cpp` 时，会生成一份 `GetInstance()` 的机器码和它独立的静态变量。
    - 编译器在编译 `B.cpp` 时，也会生成一份 `GetInstance()` 的机器码和静态变量。

  - 正常的合并机制（为什么通常没事？）

    在默认情况下（没有 `-fvisibility=hidden`），链接器（Linker）在最后打包时，看到这两个 TU 里有同名的 `inline` 函数符号，它会启动**去重机制（Deduplication）**：**只保留其中一个符号，把所有对该函数的调用都指向这同一个地址**。这样，多个 TU 最终访问的还是同一个静态变量。

  - 致命一击：`-fvisibility=hidden` 做了什么？

    当你开启了 `-fvisibility=hidden`（通常用于减少动态库大小、隐藏商业代码逻辑）时，这个参数告诉编译器：**"除非我显式声明，否则所有导出的符号（函数、类）对外部都是不可见的（Hidden）。"**

    当 `A.cpp` 和 `B.cpp` 分别属于不同的模块（例如 `A.cpp` 被编译进了主程序，而 `B.cpp` 被编译进了一个动态链接库 `.so` / `.dylib` / `.dll`）时：
    1. 链接器在打包动态库时，发现 `GetInstance()` 符号被标记为了 `hidden`。
    2. 链接器认为："哦，这个符号是内部私有的，外面绝对不需要访问它。"
    3. 于是，链接器**拒绝将动态库内部的 `GetInstance` 符号与外部主程序的符号进行合并（Weak Symbol Merging）**。

    最终结果就是：**主程序里有一份 `xxxRegistry` 实例，动态库里也死死地藏着另一份 `xxxRegistry` 实例。** 你在动态库里注册了半天，主程序去它自己的实例里找，当然空空如也！

- **解决措施**：
  - 显式将单例函数标记为强行导出（推荐）。

    既然是因为 `hidden` 导致符号没有合并，那我们就显式告诉编译器：**这个单例函数必须公开（Default 可见性），允许跨动态库合并。**

    在 GCC/Clang 下，使用 `__attribute__((visibility("default")))`。
    在 Windows (MSVC) 环境，对应的写法是跨 DLL 导出的 `__declspec(dllexport)` 和 `__declspec(dllimport)`。

    ```cpp
    class xxxRegistry {
    public:
        // 显式指定该函数具备 default 可见性，无视 -fvisibility=hidden
        __attribute__((visibility("default")))
        static xxxRegistry& GetInstance() {
            static xxxRegistry instance;
            return instance;
        }
        // ...
    };
    ```

  - 彻底从头文件中移出 GetInstance 的实现

    不要在头文件里写 `inline` 的 `GetInstance` 实现。把实现移动到**某一个特定的 `.cpp` 文件中**，并且在编译该 `.cpp` 时，确保其符号是导出的。

    ```cpp
    // ==== 头文件（xxxRegistry.h）
    class xxxRegistry {
    public:
        // 只声明，不实现（不让它变成 inline）
        static xxxRegistry& GetInstance();
    };


    // ==== 源文件（xxxRegistry.cpp，放在核心公共库中）
    #include "xxxRegistry.h"
    // 只有这一个地方有定义
    xxxRegistry& xxxRegistry::GetInstance() {
        static xxxRegistry instance;
        return instance;
    }
    ```

    这样其他各个 TU 在引入头文件时，拿到的只是一个函数的"路标"（声明），最终所有人都会老老实实通过动态链接器的符号表指向 `xxxRegistry.o` 里的唯一实例。

### 总结防坑口诀

- **单例用局部静态，无惧初始化倒装；**
- **打包静态库注意，链接优化把码伤；**
- **动态加载涉多线，加把锁儿保平安。**

## 注册工厂框架拆解

### 诉求

**原始诉求：**
给一个 "A"，运行时自动得到对应的 "B"。

```log
A  ------>  B（自动匹配 + 创建）
```

**直观写法：**
如果不用任何设计模式，可以

```log
if (A=="A1") returnnewB1();
if (A=="A2") returnnewB2();
```

可能问题：

- 每加一个 B，就要改 if-else（违反 OCP）
- 逻辑耦合严重
- 不适合插件化，不适合多人开发

**工业界解法：**

拆成三件事，标准注册机制其实就三块：

```log
[注册] + [存储] + [创建]
```

### 逐层拆解（核心）

1. Registry（注册表）------"数据库"

   它的作用：**存 A → B 的映射关系**

   类似：

   | Key (A) | Value (<font color=blue>B构造函数</font>) |
   | ------- | ----------------------------------------- |
   | "A1"    | new B1()                                  |
   | "A2"    | new B2()                                  |

   代码本质：

   ```cpp
   unordered_map<string, function<BaseB*()>> registry;
   ```

   可以理解成：**运行时字典 / 小型数据库**

   必要性：因为 C++ **没有运行时反射机制（不像 Java/Python）**，所以必须自己做一个"反射系统"。

2. Creator（工厂函数）------"怎么造 B"

   它的作用：不是直接存对象，而是存**如何创建 B**。

   错误方式：直接存对象

   ```cpp
   registry["A1"] = new B1();
   ```

   问题：
   - 对象提前创建（不灵活）
   - 生命周期不可控（手动 new 和 delete ）
   - 无法传参数

   正确方式：存构造方法

   ```cpp
   registry["A1"] = []() {
       return std::make_unique<B1>();
   };
   ```

3. Registrar（注册器）------"自动登记员"

   这是很多人最困惑的部分。

   作用：**程序启动时自动把 A1 → B1 注册进 registry**
   本质：**利用 C++ 静态初始化机制做自动注册**。程序启动时：`静态变量初始化 → 自动执行构造函数 → 注册完成`，无需在业务流程中手动注册。

   如果有 Registrar:

   ```cpp
   static Registrar r("A1", createB1);
   ```

   如果没有 Registrar，你必须手动写：

   ```cpp
   Registry::register("A1", createB1);
   ```

   问题：
   - 需手动在业务流程中注册，忘了调用就崩
   - 插件无法自动加载

4. 宏（Macro）------"减少重复代码"

   ```cpp
   REGISTER_B_CLASS("A1", B1)
   // 展开后
   static Registrar r_A1_B1("A1", [](){ return new B1(); });
   ```

   因为注册代码**非常重复**，使用宏可以**降低样板代码 + 防止写错 key**。

   不用宏你要写：

   ```cpp
   static Registrar r1("A1", [](){ return new B1(); });
   static Registrar r2("A2", [](){ return new B2(); });
   ```

### 执行顺序理解

我帮你用"执行顺序"重建一遍：

1. 程序启动

   ```cpp
   static Registrar r("A1", createB1);
   ```

2. Registrar 构造函数执行

   ```cpp
   Registry::instance().register("A1", creator);
   ```

   写入 map：`"A1" → lambda_create_B1`

3. Registry 存储完成

   ```cpp
   registry = {
       "A1": createB1,
       "A2": createB2
   }
   ```

4. 用户请求创建

   ```cpp
   auto obj = Registry::create("A1");
   ```

5. 查表 + 创建

   ```cpp
   registry["A1"]()
   ```

6. 得到对象

   ```cpp
   B1 instance
   ```

**总结：**

- Registry = 存映射
- Creator = 如何构造对象
- Registrar = 自动注册工具
- Macro = 减少重复代码

## 虚函数 ABI 兼容性与跨 SO 加载问题

**一、问题现象**

1. 二进制执行失败，GDB 调试发现，运行时在一个不期望的 `virtual` 函数上 crash
2. 通过 `ls -la lib64/` 发现部分 `.so` 时间戳不一致：包含 adapter 的 `adapter_xx.so` 已重新编译，且包含 adaptee 基类的 `adaptee_1.so`已重新编译，但包含 adaptee 基类的 `adaptee_xx2.so` 仍为旧版本
3. 问题不会在所有平台/所有用例上出现，而是**仅在旧 SO 与新主程序混合加载时出现**

**二、根因分析**

1. C++ 虚函数表（vtable）基础

   当一个 C++ 类包含虚函数时，编译器会为每个类生成一个**虚函数表（vtable）**，并在每个对象中放一个虚表指针（vptr）。子类继承父类时，编译器按**声明顺序**在 vtable 中排列函数指针。
   如果我们**在基类中间插入一个新的虚函数**，所有排在该函数后面的虚函数在 vtable 中的序号（offset）都会 +1。即使后续用新的函数覆盖（override），表的 offset 发生了变化。

2. 动态加载（dlopen/dlsym）的 ABI 不兼容

   我们的系统采用 Plugin 架构：
   - `adapter` 作为**主程序**（新编译）
   - `adaptee1.so`、`adaptee2.so`、`adaptee3.so` 作为**动态加载的子 SO**

   主程序在编译时，依赖的 `Adaptee` 基类已经生成了固定的 vtable layout。假设之前基类的虚函数顺序是：

   | offset | virtual func |
   | ------ | ------------ |
   | 0x00   | ~Adaptee     |
   | 0x08   | Compile      |
   | 0x10   | FuncReg      |
   | ...    | ...          |

   如果在 `Compile` 和 `FuncReg` 之间**插入一个新虚函数 `GetAddr`**，新主程序的 vtable 变为：

   | offset | virtual func       |
   | ------ | ------------------ |
   | 0x00   | ~Adaptee           |
   | 0x08   | Compile            |
   | 0x10   | **GetAddr** (新增) |
   | 0x18   | FuncReg            |
   | ...    | ...                |

   而旧的 `adaptee3.so` 没有重新编译，其内部的 `Adaptee` 子类 vtable 仍然是旧布局，不含 `GetAddr`。当主程序通过 vptr 调用 `FuncReg` 时，实际上访问的是偏移 `0x18`，但在旧 SO 中该位置恰好是别的函数，导致执行到**错误地址**，进而引发 `pureFunction` 崩溃或执行非预期的逻辑。

3. 为什么出错点在 CreateSubOpsStore

   `adapter` 在第一次加载子 SO 时，会调用各适配器的注册入口。由于 vtable 偏移错位，本来应该调用 `FuncReg` 的入口被重定向到别的函数上，后续再调用 `Createxxx` 时传递了错误的数据上下文，最终表现为 `Createxxx failure`。

**三、影响范围**

| 组件               | 是否受影响      | 原因                          |
| ------------------ | --------------- | ----------------------------- |
| `adapter` (主程序) | ✅ 需要重编     | 基类头文件改了，vtable 变了   |
| `adaptee1.so`      | ✅ 需要重编     | 继承 `Adaptee` 的子类         |
| `adaptee2.so`      | ✅ 需要重编     | 继承 `Adaptee` 的子类         |
| `adaptee3.so`      | ✅ **原需重编** | 旧 SO 与新主程序 ABI 不匹配   |
| 其他旧 SO          | ⚠️ 视情况       | 只要是基于旧 `Adaptee` layout |

**四、解决思路**

1. **方案一：为所有受影响 SO 强制重编**
   - 对于这种插件式架构，修改 adapter/adaptee 的虚函数机制后，所有涉及的so都要重新编译。

2. **方案二：将新增虚函数放到基类末尾（采用）**

   这是最推荐的方案，原因：
   - C++ vtable 是**前向兼容**的：只要**不对已有虚函数做重排**，旧 SO 的 vtable 虽然短一些（缺少尾部新增虚函数），但前面所有已存在的虚函数的 offset 与主程序完全对齐
   - 在主程序调用旧 SO 的方法时，所有既有接口仍通过正确的 offset 被调用
   - 新增的虚函数虽然存在于主程序的新 vtable 尾部，但旧 SO 的子类如果继承了基类，编译器已经为它生成了一个不含新增函数的版本；当主程序调用该虚函数时，会正确走到基类的默认实现或子类 override（如果未来子类重编了）

   **实现步骤**：
   1. 将 `GetAddr` 的声明从 `Adaptee` 类的中间位置移至**类末尾**
   2. 同步修改所有子类头文件，将 override 声明也移至对应类末尾，保持代码风格一致
   3. 重新编译 `adapter`、`adaptee1`、`adaptee2`
   4. `adaptee3.so` **无需重编**，因为它仍按旧的短 vtable 正常运行

3. **方案三：非虚函数替代（向下扩展 C API）**

   如果必须保持中间插入的语义，可以将新功能通过纯 C 接口或 `dlsym` 动态获取，而不是加到基类虚函数中。这种方式：
   - 不破坏 C++ vtable 稳定性
   - 但实现成本高，需要改动 Plugin 架构的接口约定
   - 适合大规模、长期维护的公共基类

1. **基类虚函数顺序不可随意变动**
   - 对含有动态加载 Plugin/Adapter 的 C++ 基类，任何在中间插入/删除虚函数的操作都是**高风险 ABI 变更**
   - 新增虚函数**务必追加到类末尾**，或在基类之外通过独立接口暴露
